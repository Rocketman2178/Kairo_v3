## Security Advisories

### ERROR Level (6)
All 6 are `security_definer_view` on read-only session/conversation views:
- `public.v_test_conversations`
- `public.organization_settings_view`
- `public.full_session_details_view`
- `public.available_sessions_view`
- `public.session_recommendations_view`
- `public.v_conversation_history`

These views use SECURITY DEFINER which enforces the view creator's RLS rather than the querying user's. Since these are read-only session/availability views containing non-sensitive public data, risk is low — but should be reviewed and recreated without SECURITY DEFINER where possible.
Reference: https://supabase.com/docs/guides/database/database-linter?lint=0010_security_definer_view

### WARN Level
**Mutable search_path (27 functions):** Functions without `SET search_path = ''` are vulnerable to search_path injection. Affected functions include core ones: `confirm_registration`, `create_pending_registration`, `get_pending_registration`, `add_to_waitlist_with_position`, `check_session_availability`, `get_matching_sessions`, `get_alternative_sessions`, `calculate_registration_discount`, and others. Per CLAUDE.md, all new functions must include `SET search_path = ''`. Existing functions should be patched in a migration.

**Overly permissive RLS policies (5):**
| Table | Policy | Issue |
|---|---|---|
| `competitive_recommendations` | `Authenticated users can manage competitive recommendations` | `USING (true)` + `WITH CHECK (true)` for ALL — any authenticated user can read/write all rows |
| `conversations` | `Anyone can create conversations` | `WITH CHECK (true)` INSERT for public — intentional per CLAUDE.md pattern, acceptable |
| `regions` | `Authenticated users can manage regions` | `USING (true)` for ALL — any authenticated user can manage all regions |
| `seasons` | `Authenticated users can manage seasons` | `USING (true)` for ALL — any authenticated user can manage all seasons |
| `tiger_tank_feedback` | `Anyone can submit feedback` | `WITH CHECK (true)` INSERT for anon+authenticated — intentional, acceptable |

The `competitive_recommendations`, `regions`, and `seasons` tables should have organization-scoped policies, not `USING (true)` for all authenticated users.

## Issues Found & Resolved

### Critical (Fixed)
1. **`Register.tsx` — wrong `families` column name**: All `families` inserts and the email lookup used `name` instead of `primary_contact_name`. Every new family creation and returning family lookup would fail with a 400/422 error. **Fixed in commit `f2f97d0`.**

2. **`Register.tsx` — wrong `children` column name**: `medical_notes` (text) does not exist; actual column is `medical_info` (jsonb). Every child insert would fail. **Fixed in commit `f2f97d0`.**

3. **`Register.tsx` — null `date_of_birth` violates NOT NULL**: When user skips date of birth, `null` was inserted into a NOT NULL column. Now defaults to today's date. **Fixed in commit `f2f97d0`.**

### Critical (Outstanding — Not Fixed in This Session)
4. **`stripe-webhook` edge function — `payments` insert missing `family_id`**: `payments.family_id` is NOT NULL. The webhook inserts a payment row without this field. Will cause silent payment confirmation failures when Stripe is live. Requires fix in edge function before Stripe go-live.

### Non-Critical (Outstanding)
5. **`create-payment-intent` — sibling discount still 10%**: Hardcoded at `discountPercent = 10` instead of 25%. Frontend correctly applies 25% via `discountCalculator.ts`, but when Stripe is live the server-side intent will be calculated at the lower amount, causing a price discrepancy.

6. **Over-capacity seed data**: One session (Premier Soccer, 2027-12-07) has `enrolled_count=18 > capacity=12`. This is test seed data with `status="full"` and does not affect production logic, but the data inconsistency should be corrected in a cleanup migration.

7. **`RegistrationConfirmation.tsx` hardcoded soccer copy**: "Soccer cleats optional but recommended" is hardcoded in the before-session checklist. This should be organization-specific or program-type-specific, not hardcoded.
