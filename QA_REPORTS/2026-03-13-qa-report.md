## Security Advisories

### ERROR-Level (Resolved)
- ✅ `public.regions` — RLS disabled with policies present → **FIXED**: RLS enabled via migration
- ✅ `public.competitive_recommendations` — RLS disabled → **FIXED**: RLS enabled + policies added

### ERROR-Level (Pre-existing, not newly introduced)
- ⚠️ `public.v_test_conversations` — SECURITY DEFINER view (intentional for test data access)
- ⚠️ `public.organization_settings_view` — SECURITY DEFINER (intentional)
- ⚠️ `public.full_session_details_view` — SECURITY DEFINER (intentional — Kai needs cross-org read)
- ⚠️ `public.available_sessions_view` — SECURITY DEFINER (intentional — Kai needs cross-org read)
- ⚠️ `public.session_recommendations_view` — SECURITY DEFINER (intentional)
- ⚠️ `public.v_conversation_history` — SECURITY DEFINER (intentional)

> These SECURITY DEFINER views are intentional — Kai reads session data without per-user RLS constraints. Accept as-is.

### WARN-Level (Open)
- ⚠️ 27 functions with mutable `search_path` — low-risk in controlled environments, consider adding `SET search_path = public` to production-critical functions like `confirm_registration`, `create_pending_registration`, `get_pending_registration`
- ⚠️ `conversations.INSERT` policy with `WITH CHECK (true)` — intentional for anonymous chat creation
- ⚠️ `seasons` ALL policy with `USING(true)/WITH CHECK(true)` for authenticated — any authenticated user can manage all seasons; acceptable for single-tenant demo but should be scoped to org_id in multi-tenant production
- ⚠️ `tiger_tank_feedback.INSERT` with `WITH CHECK(true)` — intentional for anonymous feedback submission

## Vulnerabilities Tested

| Attack Vector | Protected? | Details |
|---|---|---|
| SQL Injection (DB layer) | YES | Supabase uses parameterized queries; all RPC functions accept typed parameters |
| SQL Injection (application layer) | YES | No string concatenation in SQL; Supabase client parameterizes all queries |
| XSS (frontend) | YES | React renders text as text nodes, not innerHTML; no dangerouslySetInnerHTML usage found |
| Auth Bypass (RLS) | YES | All 10 critical tables now have RLS enabled post-fix |
| Payment Forgery (create-payment-intent) | PARTIAL | Protected by registrationToken; no JWT auth; acceptable risk for current stage |
| Webhook Forgery (stripe-webhook) | CONDITIONAL | Signature validation only when STRIPE_WEBHOOK_SECRET is set; must be set in production |
| CORS Bypass | N/A | CORS is `*` (acceptable for public API), no sensitive data accessible without auth |
| Data Enumeration via anon role | PASS | Tested check_session_availability with fake UUID — returns error, not data leak |
| Registration Token Brute Force | LOW RISK | Token is UUID v4 (2^122 entropy); 15-minute expiry via `expires_at` |
